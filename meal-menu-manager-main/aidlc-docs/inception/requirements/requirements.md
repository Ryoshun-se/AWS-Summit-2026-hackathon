# 要件定義書

## アプリコンセプト

### ビジョン: 食事に関する選択行為をゼロにする — 「人をダメにする」アプリ

**コンセプトアピール**:
現代人は 1 日に最大 35,000 回の意思決定を行うと言われています。その多くは「今日の夕食は何にしよう」「冷蔵庫に何が残っているか」「子供は食べられるか」「先週も食べたっけ」といった、食事にまつわる無数の小さな選択です。これらの選択は一つひとつは些細でも、積み重なることで脳のリソースを消耗させ、本当に重要な判断の質を下げる「選択疲れ（Decision Fatigue）」を引き起こします。

ここで重要なのは、食事における大変さの本質は **調理そのものではない** という点です。多くの人が疲弊しているのは、包丁を握ることでも火を使うことでもなく、**「何を作るか」を決めること** と **「何を・どこで・いくらで買うか」を調べること** の 2 つです。メニューを考える際には、過去に食べたもの・家族の好み・栄養バランス・旬の食材・冷蔵庫の残り物・食費予算など、膨大な変数を頭の中で処理しなければなりません。そしてメニューが決まった後も、スーパーのチラシを確認し、どの店が安いかを比較し、買い忘れがないようにリストを作るという、もう一段階の意思決定が待っています。

しかもこの意思決定は、どれだけ上手くこなしても誰にも褒められることはありません。栄養バランスが取れていて、家族全員が満足して、食材の無駄もなく、予算内に収まっていたとしても、それは「当たり前」として消費されるだけです。報われない意思決定を、毎日、何年も繰り返す。それが食事にまつわる現実です。

このアプリのコンセプトは **「人をダメにする」** です。

AI がすべての制約を満たした献立を自動生成し、食材を集計し、将来的には発注まで完結させる。その結果、栄養バランスは保たれ、家族の不満は出ず、食費も予算内に収まる。完璧にこなされた意思決定が毎週届き続けると、人はやがて気づきます。**「自分でやる必要が、どこにもない」** と。

大変で、複雑で、誰にも評価されない意思決定を一度アプリに委ねてしまえば、二度と自分でしようとは思わなくなるでしょう。献立を自分で考える能力は、使われないまま静かに錆びついていくかもしれません。それこそが、このアプリが目指す「ダメにする」の本質です。

かつて「人をダメにするソファ」が身体を預けることで究極のリラックスを提供したように、このアプリは **メニュー検討と食材調達という 2 つの意思決定を丸ごと引き受ける** ことで、ユーザーの脳と時間を、そして食事への執着を、静かに解放します。

**食事に関するすべての選択をアプリに委ねてください。あなたはただ、食べるだけでいい。**

## ビジネスモデル

### 収益モデル: BtoC フリーミアム + 仲介料・広告収入

| 収益源 | 内容 |
|---|---|
| **サブスクリプション（フリーミアム）** | 無料プランで基本機能を提供し、プレミアムプランで高度な AI パーソナライズ・詳細な栄養管理・複数プロファイル等を提供 |
| **連携スーパー・ネットスーパーからの仲介料** | 将来フェーズで実装する食材発注機能において、連携先スーパーからの仲介手数料を収益化 |
| **広告収入** | 食品メーカー・スーパー・調理器具メーカー等からのターゲティング広告（プレミアムプランでは非表示） |

### フリーミアム設計方針（将来フェーズで詳細化）
- **無料プラン**: 週間献立生成（基本）、買い物リスト、食事履歴記録
- **プレミアムプラン**: 詳細な栄養素管理、複数プロファイル（家族全員分）、ネットスーパー連携、広告非表示、マイレシピを無制限に登録

## ターゲットユーザー

このアプリは「食事の選択行為に疲れているすべての人」を対象とします。ライフスタイルや健康目標は異なっても、**「食事の意思決定を減らしたい」** という共通ニーズを持つ幅広いユーザー層をカバーします。

| ペルソナ | 主な課題 | アプリへの期待 |
|---|---|---|
| 子育て中の親 | 家族全員が食べられる献立を毎日考えるのが大変 | 子供対応・栄養バランスを自動で考慮した献立 |
| 共働き夫婦 | 仕事後に献立を考える余力がない | 何もしなくても週の献立が決まっている状態 |
| ボディメイク実践者 | PFC バランスを保ちながらバリエーションを出すのが難しい | 筋トレ・リカバリーを考慮した高タンパク献立の自動生成 |
| 生活習慣病のシニア | 塩分・脂質制限の中で飽きない献立を考えるのが大変 | 医療的制約を守りながら多様な献立を自動生成 |
| 一人暮らしの学生・社会人 | 食材が余る・栄養が偏る・食費が限られる | 1 人分・予算内・栄養バランスの取れた献立 |

## 開発フェーズ戦略

このプロジェクトは段階的に開発します。

### PoC フェーズ（本要件定義の対象）
- 献立生成 + 買い物リスト自動生成
- AI パーソナライズ・学習機能
- ユーザー確認ポイント（献立確認）の実装

### 将来フェーズ（PoC 完成後）
- ネットスーパー API 連携（カート自動追加 + 注文自動確定）
- 発注前の最終確認機能
- 価格比較機能（複数ネットスーパー対応）

## 機能要件

### PoC フェーズ スコープ

### FR-01: ユーザー認証・アカウント管理
- メールアドレス＋パスワードによるユーザー登録・ログイン・ログアウト
- パスワードリセット機能
- ユーザーごとのデータ分離（マルチユーザー対応）

### FR-02: ユーザープロファイル設定
ペルソナに関わらず共通のプロファイル構造を使用し、ライフスタイルモードの選択によって AI への制約条件をカスタマイズする。

**共通設定**:
- 食事制限・アレルギー情報の登録（グルテンフリー、乳製品不使用、低塩分、低脂質など）
- 好みの料理ジャンル設定（和食、洋食、中華など）
- 生成対象の食事区分（朝食・昼食・夕食の任意の組み合わせ）
- 家の定番料理・常備菜の登録

**ライフスタイルモード**（いずれか 1 つを選択、AI への制約条件が切り替わる）:
- **ファミリーモード**: 子供が食べられる料理・家族全員対応・食材バランス重視
- **ボディメイクモード**: PFC（タンパク質・脂質・炭水化物）バランス目標値の設定、筋トレ後リカバリー食の考慮、高タンパク食材の優先
- **ヘルスケアモード**: 塩分・脂質・糖質などの医療的制約値の設定、制約内でのバリエーション最大化
- **ソロモード**: 1 人分の食材量最適化、食材廃棄最小化、食費予算設定、栄養バランス重視

> ※ PoC フェーズでは単一モード選択のみ対応。複数モードの組み合わせ（例：塩分制約＋PFC 管理、ファミリー＋ヘルスケアなど）は将来フェーズで検討。

**健康目標**（ライフスタイルモードと組み合わせ可能）:
- ダイエット、筋肉増強、血糖値管理、血圧管理、体重維持など

### FR-03: 献立生成設定
- 生成対象の食事区分を設定可能（朝食・昼食・夕食の任意の組み合わせ）
  - 例：夕食のみ、朝食＋夕食、全食事など
- 生成期間は１週間単位

### FR-04: AI による献立自動生成
- ユーザープロファイル（食事制限、健康目標、好みのジャンル）に基づいた献立生成
- 直近に食べた料理の履歴を考慮した献立生成（重複回避・バランス調整）
  - 重複回避の対象は、アプリが提案した料理だけでなく、実際に食べた料理（外食・買い食い・献立変更を含む）すべて
  - 即時リカバリー: 食事履歴が記録された時点（外食・買い食い・献立変更を含む）で、残り週内の献立を即時再調整する（次週の献立生成まで待たない）。リカバリーの観点はモードごとに異なる（FR-04-1 参照）
- 子供が食べられる料理を優先的に提案する設定への対応
- 家の定番料理・常備菜を学習し、定期的に献立に組み込む
- AWS Bedrock または OpenAI API を使用した AI 生成
- **選択行為ゼロの原則**: ユーザーが何もしなくても毎週自動で献立が生成される（プッシュ通知で確認を促す）

#### FR-04-1: 献立生成における考慮要素（AI への制約条件）
AI は以下の要素を総合的に考慮して献立を生成する：

**共通制約（全ペルソナ）**
- 週内での野菜カテゴリのバランス（葉物・根菜・その他を偏りなく使用）
- 肉・魚のバランス（豚肉・鶏肉・牛肉・魚介類を週内で均等に配分）
- 旬の食材を優先的に使用する
- 直近 1 週間以内に食べた料理・食材の重複を避ける
- 先週の献立との重複チェック（同じ料理が連続しない）

**ファミリーモード固有制約**
- 子供が食べられる料理・食材かどうかの判定
- 家族全員が食べられる料理を優先する

**ボディメイクモード固有制約**
- 設定した PFC 目標値（タンパク質・脂質・炭水化物のグラム数または比率）に近い献立を生成
- 筋トレ後のリカバリーを考慮した高タンパク・適度な炭水化物の食事を提案
- 同じ食材でも調理法を変えてバリエーションを確保（鶏胸肉の調理法を毎回変えるなど）

**ヘルスケアモード固有制約**
- 設定した医療的制約値（塩分 g/日、脂質 g/日、糖質 g/日など）を超えない献立を生成
- 制約の中で食材・調理法のバリエーションを最大化する
- 同じ制約でも飽きない献立を提案する

**ソロモード固有制約**
- 1 人分の食材量に最適化（食材廃棄を最小化）
- 複数の料理で同じ食材を使い回せる献立を優先（例：キャベツを使う料理を週内で複数提案）
- 設定した食費予算内に収まる食材を優先
- 外食が多い場合は自炊日の栄養バランスを補完する献立を提案

**全モード共通：即時リカバリーの観点**

食事履歴が記録された時点（献立通りに食べた場合を除く）で、残りの週内の献立を即時再生成する。モードによってリカバリーの観点が異なる:

| モード | リカバリーの観点 |
|---|---|
| ファミリー | 冷蔵庫の余り食材を優先使用、栄養バランスを残りの食事で補完 |
| ボディメイク | 超過・不足した PFC を週内で帳尻合わせ（飲み会・外食含む） |
| ヘルスケア | 医療的制約値（塩分・脂質・糖質）の超過分を残りの食事で調整 |
| ソロ | 余り食材優先、食費予算内に収まるよう残り献立を調整、食材廃棄最小化 |

### FR-05: 献立確認・カスタマイズ（ユーザー確認ポイント①）
- 週間カレンダー形式での献立表示
- **必須承認**: ユーザーが明示的に「確定」操作を行うまで献立は確定しない
- 完全カスタマイズ機能（任意）：
  - 特定の日・特定の食事区分の料理を個別に再生成
  - 料理の手動入れ替え（ドラッグ＆ドロップ等）
  - 週全体の一括再生成
- 献立の保存・履歴管理

### FR-06: レシピ詳細表示
- 各料理のレシピ詳細（材料・分量・調理手順）の表示
- レシピは AI 生成または外部レシピデータと連携

### FR-07: 買い物リスト自動生成
- 週間献立から必要な食材を自動集計
- 食材ごとの必要量を表示
- 買い物リストのチェック機能（購入済みマーク）
- **PoC フェーズ**: 買い物リストの表示・手動チェックまで
- **将来フェーズへの橋渡し**: ネットスーパー連携のためのデータ構造を設計段階から考慮する

### FR-08: 食事履歴の記録・学習
- 実際に食べた料理の記録（献立通りに食べたか、変更したかを記録）
- 食事履歴は即時反映される（全モード共通）
  - 献立通りに食べた場合: 記録のみ
  - 献立変更・外食・買い食いがあった場合: 残り週内の献立を即時再生成
- 食事履歴を AI 生成に反映（直近の食事を考慮した献立提案）
- 食事履歴は週次サイクルの次回献立生成にも引き続き反映される（長期学習）
- 定番料理・常備菜の学習（ユーザーが繰り返し登録・食べた料理を学習）

## 非機能要件

### NFR-01: プラットフォーム
- **対象**: iOS のみ
- **最小サポートバージョン**: iOS 16 以上（PoC フェーズでは最新版を対象）

### NFR-02: 技術スタック（暫定）
- **フロントエンド**: 要件確定後に選定（iOS ネイティブ Swift または Flutter を候補とする）
- **バックエンド**: AWS（Lambda + API Gateway）
- **AI**: AWS Bedrock または OpenAI API
- **データベース**: AWS（DynamoDB または RDS — 設計フェーズで決定）
- **認証**: AWS Cognito または Firebase Authentication（設計フェーズで決定）

### NFR-03: スケール・パフォーマンス
- **想定ユーザー規模**: 個人・家族向け PoC（〜数十ユーザー）
- **献立生成レスポンス**: 10 秒以内（AI 生成を含む）
- **API レスポンス**: 通常操作は 2 秒以内

### NFR-04: セキュリティ
- セキュリティ拡張ルール: **無効**（PoC フェーズのため）
- 基本的な認証・認可は実装する（JWT トークン等）
- ユーザーデータの基本的な分離を保証

### NFR-05: オフライン対応
- **不要**（常時インターネット接続を前提とする）

### NFR-06: テスト方針
- プロパティベーステスト（PBT）拡張ルール: **無効**（PoC フェーズのため）
- 基本的なユニットテストおよび統合テストを実装

### NFR-07: 保守性・拡張性
- PoC フェーズのため、シンプルな構成を優先
- 将来的な Android 対応・ユーザー規模拡大を考慮した設計（過度な先行投資は不要）

## ユーザーシナリオ（主要フロー）

### シナリオ 1: 初回セットアップ
1. ユーザーがアプリをインストールし、メールアドレスで登録
2. プロファイル設定（食事制限、健康目標、好みのジャンル、子供の有無、定番料理）
3. 生成対象の食事区分を設定（例：夕食のみ）
4. 初回の週間献立を AI が自動生成
5. プッシュ通知で献立確認を促す

### シナリオ 2: 週次の自動サイクル（選択行為ゼロの理想フロー）
1. 毎週決まったタイミングで AI が自動的に週間献立を生成
2. プッシュ通知でユーザーに確認を促す（**ユーザー確認ポイント①**）
3. ユーザーが確認・必要に応じてカスタマイズし、「確定」ボタンで明示的に承認する
4. 確定した献立から買い物リストを自動生成
5. **PoC フェーズ**: 買い物リストを確認し、購入済みをチェック
6. **将来フェーズ**: 発注前の最終確認（**ユーザー確認ポイント②**）→ 自動発注

### シナリオ 3: 献立カスタマイズ（任意）
1. ユーザーが週間カレンダーで献立を確認し、気になる料理を AI に伝える
2. AI が代替案を自動生成し、献立を更新する
3. 更新された献立に合わせて買い物リストが自動反映される

### シナリオ 4: 食事履歴の記録
1. 食事後、実際に食べた料理を記録（献立通り or 変更）
2. 履歴が蓄積され、次回の献立生成に反映される
3. 定番料理・常備菜として学習が進む

### シナリオ 5: 即時リカバリー（新規追加）
1. ユーザーが外食・買い食い・献立変更を食事履歴として記録
2. アプリが残り週内の献立を即時再生成（モードごとの観点でリカバリー）
3. 再生成された献立に合わせて買い物リストも自動更新
4. プッシュ通知で変更を知らせる
